\chapter{Planung und Konzeption}

\section{Systemarchitektur und Datenfluss}
 Der Weg der Daten durch die OSI-Schichten. Sensor -> TCP/MQTT (WLAN) -> Raspberry Pi -> TCP/HTTP (Docker Bridge) -> Datenbank. Später dann: Grafana -> Cloudflare Tunnel -> Endnutzer.

\section{Netzwerktopologie}
Geplant war eine autarke Sterntopologie mit dem Raspberry Pi als zentralem Access Point für alle Geräte.

\section{Aufbau des Sensors}

Für den praktischen Aufbau des Systems werden folgende Komponenten benötigt:

\begin{itemize}
	\item DHT11 - Temperatur- und Luftfeuchtigkeitssensor
	\item ESP32 - Mikrocontroller mit WLAN-Funktion
	\item Breadboard - zur schnellen, lötfreien Prototypenentwicklung
	\item Micro-USB-Kabel - zur Stromversorgung und Programmierung
	\item 3 Jumper-Kabel - zur elektrischen Verbindung zwischen Sensor und Mikrocontroller
\end{itemize}

Der Aufbau erfolgt wie in Abbildung 3.1 dargestellt. 
Der DHT11 wird mit drei Leitungen an den ESP32 angeschlossen:
ein Datenpin, ein 3.3-V-Anschluss sowie eine Masseverbindung. 
Das Micro-USB-Kabel verbindet den ESP32 mit einem Computer und
dient sowohl als Stromquelle als auch zur Übertragung des
Programmcodes. Durch die geringe Leistungsaufnahme des 
Systems reicht die USB-Stromversorgung vollständig aus.

Zur Programmierung wird Visual Studio Code mit der Erweiterung 
PlatformIO genutzt, die Mikrocontrollerprojekte, 
Bibliothekenverwaltung, Kompilieren und Hochladen auf den ESP32 
vereinfacht. Für eine einfache Weboberfläche zur Anzeige der 
Messdaten kommt zusätzlich Live Server zum Einsatz, um die 
Website lokal zu hosten und in Echtzeit zu aktualisieren.

Der Plan sieht vor, den DHT11 in regelmäßigen Abständen 
auszulesen, die Temperatur- und Feuchtigkeitswerte digital 
zu verarbeiten und über den seriellen Monitor zu prüfen. 
Optional können die Daten über die Weboberfläche dargestellt 
werden, um die Genauigkeit des Sensors für Anwendungen wie 
Defogging-Systeme zu beurteilen.
